# Бриф для Claude Design — платформа «Своя игра»

> Это бриф для подготовки визуального дизайна платформы. Он самодостаточен: всё, что
> нужно знать о продукте, ролях и экранах, — ниже. Реализация — web (Svelte), но дизайн
> нужен на уровне визуального языка, экранов и компонентов, не кода.

---

## 1. Что это и для кого

«Своя игра» — домашняя командная игра-викторина (наследие одноимённого ТВ-шоу, по духу
близко к Jeopardy!). Семья/друзья собираются дома, ведущий запускает игру, команды
отвечают на вопросы из сетки категорий разной стоимости. Всё работает в локальной сети
(LAN): один компьютер-сервер, остальные подключаются с телефонов и большого экрана.

Это **не коммерческий продукт**, а тёплый домашний праздник: вечер с друзьями, азарт,
смех, лёгкое соперничество. Дизайн должен поддерживать ощущение «шоу», но не быть
кричаще-рекламным.

### Роли и устройства (три очень разных контекста)

| Роль | Устройство | Дистанция/поза | Суть |
|---|---|---|---|
| **Игрок** | телефон | в руке, одной рукой, быстро | главное — кнопка «жму» (buzzer); реакция критична по времени |
| **Табло** | большой экран / ТВ / проектор | смотрят через комнату, 2–5 м | только просмотр; драматично, крупно, читаемо издалека |
| **Ведущий** | ноутбук | сидит, кликает быстро по ходу игры | плотная панель управления; нельзя промахнуться |
| **Гость (лендинг)** | телефон/ноутбук | заходит впервые | увидеть активную игру, войти |
| **Админ (редактор)** | ноутбук | работает вдумчиво, заранее | продуктивный инструмент: база вопросов + конструктор игры |

Дизайн должен дать **один узнаваемый визуальный язык**, который по-разному раскрывается
в этих контекстах: зрелищно на табло, моментально-понятно на телефоне игрока,
эффективно-плотно в инструментах ведущего и админа.

---

## 2. Доменные понятия (важно для специфики)

- **Игра** состоит из **раундов**. Раунд — **прямоугольная сетка**: строки —
  **категории** (темы), столбцы — **стоимости** (цена растёт слева направо). Ячейка =
  один вопрос данной категории за данную цену.
- **Вопрос** имеет тип: текст / картинка / аудио. У него есть текст вопроса и ответ
  (ответ видит только ведущий).
- **Мини-игры** — особые ячейки, помечаются иконкой:
  - **«Кот в мешке»** (иконка кота) — вопрос передаётся другой команде.
  - **«Аукцион»** (иконка весов/торга) — команды делают ставки.
- **Команды** и **игроки**: игрок вводит имя/фамилию, входит в команду; у команды есть
  счёт (может быть отрицательным).
- **Buzzer** («жму!») — у игроков. Кто быстрее нажал после открытия — отвечает. Есть
  фальстарт (нажал раньше — временный блок) и «змейка» очереди ответивших.

Эти понятия — источник **сигнатурных** элементов дизайна (см. §6): сетка цен, кнопка
buzzer, табло-счёт, иконки мини-игр.

## 3. Карта экранов

### A. Публичный лендинг — `/` (гость, телефон/ноутбук)
- **Задача:** показать активную игру и дать войти.
- Список игр (обычно одна активная). По игре видно: **название**, **макс. число команд**,
  **сколько игроков уже в игре**, **всего раундов** и **какой раунд идёт сейчас**.
- Состояния: есть активная игра / нет активных игр (пустой экран — приглашение, не
  «ошибка») / игра завершена (приглушённая).

### B. Вход игрока — `/play?game=…` (телефон)
- **Задача:** быстро представиться и попасть в команду.
- Поля: фамилия, имя; выбрать существующую команду **или** создать новую.
- Состояния: форма входа / игра не найдена / ожидание ведущего.

### C. Экран игрока — `/play` в игре (телефон, ГЕРОЙ = buzzer)
- **Главный момент всей платформы** — кнопка buzzer: огромная, мгновенно нажимаемая
  большим пальцем, очевидное состояние (готова / нажата / заблокирована с таймером
  фальстарта).
- Другие состояния: «ВЫ ОТВЕЧАЕТЕ!», «ВЫБИРАЙТЕ ВОПРОС», показ текста вопроса, ожидание.
- **Таймер ответа:** когда команда отвечает, на её экране — крупный обратный отсчёт
  (заданные ведущим 30–60 с). Визуально нарастающая тревога к нулю; по истечении ответ
  считается неверным и ход уходит — экран это явно показывает.
- Минимум текста, максимум ясности «что от меня сейчас нужно».

### D. Табло — `/board?game=…` (ТВ/проектор, только просмотр)
- **Задача:** зрелищно показать состояние всем в комнате.
- Виды: заставка «игра ещё не началась»; **сетка раунда** (категории × цены, отыгранные
  ячейки гаснут); **крупный вопрос** (текст/картинка/аудио) во весь экран; **очередь
  buzzer**; **счёт команд**; итоги раунда; финал-победитель.
- Всё читаемо с 2–5 метров: крупная типографика, высокий контраст, мало мелочей.

### E. Экран ведущего — `/host` (ноутбук, плотная панель)
- **Задача:** управлять живой игрой без промахов и пауз.
- Настройка игры (выбор готовой игры, число команд, **таймер ответа 30–60 с**) и резюме
  прерванной игры.
- Лобби: управление командами (создать/переименовать/удалить), ростер игроков
  (перемещение между командами, индикатор онлайн/офлайн).
- Игра: та же сетка для выбора вопроса; текст вопроса **и ответ** (ответ — только
  ведущему); контролы по типу (обычный: зарядить/открыть buzzer/судить верно-неверно;
  аукцион: ставки команд, объявить победителя; кот: выбрать команду-получателя);
  очередь buzzer; правка очков (±цена, произвольная дельта); конец раунда/игры.
- **Таймер ответа** виден и ведущему — тот же обратный отсчёт по отвечающей команде, с
  возможностью понимать, что время вышло (ход ушёл, команде засчитан неверный ответ).
- Нужны **крупные, разнесённые, явно подписанные** действия; критичные (Верно/Неверно)
  визуально однозначны.

### F. Логин админа — `/admin` (ноутбук)
- Простая форма: одно поле пароля, кнопка «Войти». Ошибка — понятная, в голосе
  интерфейса. После входа — оболочка (G).

### G. Админ-оболочка — `/admin` после входа (ноутбук)
- Три вкладки: **«Запустить игру»** (пока заглушка-плейсхолдер), **«База вопросов»** (H),
  **«Конструктор игры»** (I). Кнопка «Выйти».

### H. Менеджер базы вопросов (ноутбук, продуктивный инструмент)
- **Задача:** создавать и хранить контент — категории и вопросы/ответы. Это фундамент.
- Слева — список/дерево **категорий** (создать, переименовать, удалить, порядок).
- Справа — **вопросы выбранной категории** (карточки): создать/править/удалить, порядок.
- **Форма вопроса:** тип (текст/картинка/аудио), текст вопроса, ответ; для медиа —
  загрузка файла с превью (картинка/аудио-плеер). Здесь **нет** цены и мини-игр — это
  чистый контент.
- Импорт/экспорт базы.
- Без явной кнопки «сохранить» — автосохранение (нужен ненавязчивый индикатор «сохранено»).

### I. Конструктор игры (ноутбук, drag&drop — самый сложный экран)
- **Задача:** собрать игру из сетки, цен и вопросов базы.
- Игра = название + раунды (вкладки).
- Раунд = **прямоугольная сетка**: заголовки-столбцы с **редактируемой ценой**; строки —
  категории. По умолчанию заготовка 5×5; столбцы и строки добавляются/убираются.
- **Боковая панель** = дерево категорий базы (мини-карточки). Взаимодействие drag&drop:
  1. перетащить **категорию** на пустую строку → строка привязывается к категории, список
     вопросов фильтруется по ней (мини-превью вопросов);
  2. перетащить **вопрос** в ячейку — **только в строку своей категории** (в чужой ряд
     нельзя — нужна понятная визуальная обратная связь о запрете);
  3. в заполненной ячейке — переключатель **тега мини-игры** (кот / весы-аукцион / нет).
- **Лайв-валидация:** пустые/невалидные ячейки и строки подсвечены; видно, что мешает
  опубликовать.
- Действия: индикатор автосохранения, «Скачать шаблон», «Опубликовать», «Сыграть
  тестовую».
- Это **информационно-плотный** экран: сетка + панель источника + статусы. Drag&drop
  должен быть очевидным (что можно тащить, куда можно бросить, что запрещено).

## 4. Ключевые состояния, которые легко забыть

- **Пустые состояния:** нет активных игр; пустая база; новая игра без вопросов в ячейках.
  Каждое — приглашение к действию, не тупик и не «ошибка».
- **Ошибки:** неверный пароль; невалидная игра при публикации; недопустимое имя команды;
  попытка бросить вопрос в чужой ряд. В голосе интерфейса: что не так и что сделать.
- **Загрузка/ожидание:** «подключение к игре», «ждём ведущего».
- **Офлайн/реконнект:** индикатор связи игрока/команды.
- **Фальстарт buzzer:** заблокированное состояние с обратным отсчётом.
- **Таймер ответа:** активный отсчёт у ведущего и отвечающей команды; момент истечения
  (время вышло → неверно, ход ушёл). Опционально отсчёт можно дублировать на табло для
  драматизма.

---

## 5. Ключевые флоу

1. **Игрок:** лендинг → выбрал игру → представился, вошёл в команду → ждёт → выбирает
   вопрос (если его очередь) → жмёт buzzer → отвечает.
2. **Ведущий:** логин админа → «Запустить игру» (позже) или из готовой игры → лобби,
   команды → ведёт раунды (выбор вопроса, buzzer, судейство, мини-игры, правка очков) →
   итоги → финал.
3. **Автор контента:** логин → «База вопросов» → создаёт категории и вопросы (+медиа).
4. **Сборка игры:** «Конструктор игры» → сетка, цены → перетаскивает категории в строки,
   вопросы в ячейки, ставит мини-игры → валидно → «Сыграть тестовую» / «Опубликовать».

## 6. Сигнатурные элементы предметной области

Эти вещи специфичны для «Своей игры» — на них стоит строить идентичность, а не на
дженерик-паттернах:

- **Сетка цен** (категории × стоимости) — и игровая, и редакторская. Центральный
  визуальный мотив платформы. Отыгранная ячейка должна «гаснуть» осмысленно.
- **Buzzer** — кнопка-герой на телефоне. Тактильность, мгновенность, азарт.
- **Табло счёта** — командный счёт как на ТВ-шоу; лидер выделен; отрицательные значения
  читаемы.
- **Иконки мини-игр** — «кот в мешке» и «аукцион»: узнаваемые, играющие на интригу.
- **Драма открытия вопроса** — момент, когда вопрос раскрывается на весь экран табло.

## 7. Визуальное направление (открытый бриф)

Палитра и типографика **не заданы** — это твоя зона. Но есть рамки:

- **Настроение:** игра-викторина, праздник, азарт, тепло домашнего вечера. Зрелищно, но
  не аляповато; современно, но не корпоративно-стерильно.
- **Текущая тема — только референс, не обязательство:** сейчас платформа в тёмном
  неон-аркадном стиле (тёмный фон, неон-циан/маджента/золото, Segoe UI), стили инлайновые
  и несистемные. Можно развить эту линию, а можно предложить своё — но осознанно и
  обоснованно.
- **Избегай шаблонных AI-дефолтов.** Не уходи по инерции в один из трёх затёртых
  пресетов: (1) кремовый фон + высококонтрастный серив + терракота; (2) почти-чёрный фон
  + один кислотный акцент; (3) «газетная» вёрстка с волосяными линиями и нулевым
  скруглением. Если приходишь к одному из них — останься, только если это осознанный
  выбор под этот бриф, а не дефолт.
- **Один смелый акцент.** Сигнатурный элемент (см. §6, вероятнее всего — сетка или
  buzzer) — место для смелости; остальное держи дисциплинированным и тихим.
- **Типографика несёт характер.** Дисплейная гарнитура для шоу-моментов (табло, открытие
  вопроса, заголовки) + спокойная читаемая для интерфейса/плотных панелей. Не одно
  семейство «на всё».
- **Мотивы движения — по делу.** Открытие вопроса, нажатие buzzer, смена счёта — точки,
  где анимация уместна. Не рассыпай эффекты ради эффектов.

## 8. Ограничения качества (обязательный пол)

- **Язык интерфейса — русский.** Тон — дружелюбный, на «вы» по умолчанию, без канцелярита.
  Кнопки — активным глаголом, действие сохраняет имя через весь флоу («Опубликовать» →
  «Опубликовано»). Имена — по тому, чем управляет человек, не по тому, как устроена система.
- **Три радикально разных вьюпорта:** телефон (одна рука, большой палец), ТВ/проектор
  (читаемость с 2–5 м, возможно тусклая комната), ноутбук (плотные инструменты).
  Адаптивность обязательна.
- **Контраст и читаемость** — особенно табло и buzzer.
- **Доступность:** видимый фокус с клавиатуры; уважение `prefers-reduced-motion`; цель
  нажатия на телефоне крупная.
- **Тёмное окружение:** игру часто ведут при приглушённом свете — экраны не должны
  слепить и не должны тонуть.

## 9. Что нужно на выходе от Claude Design

1. **Токен-система:** палитра (4–6 именованных hex), типографика (2–3 роли: дисплейная,
   текстовая, утилитарная), шкала отступов, радиусы, тени/свечения.
2. **Сигнатурный элемент:** что делает платформу узнаваемой, с проработкой.
3. **Макеты ключевых экранов:** как минимум — табло (сетка + открытый вопрос + счёт),
   экран игрока (buzzer + состояния), конструктор игры (сетка + drag&drop), менеджер базы,
   панель ведущего в игре, лендинг. Желательно с ключевыми состояниями.
4. **Инвентарь компонентов:** кнопки (вкл. критичные верно/неверно и buzzer), карточки
   (категория, вопрос, ячейка сетки), поля и формы, табло счёта, теги мини-игр, вкладки,
   модалки (напр. выбор «новый пак / перезаписать» при публикации), индикатор
   автосохранения, пустые/ошибочные состояния.
5. **Краткое обоснование** ключевых решений (почему эта палитра/гарнитура/мотив — под
   этот бриф, а не вообще).
