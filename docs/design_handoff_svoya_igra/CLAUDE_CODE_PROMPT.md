```
Ты помогаешь реализовать продукт «Своя игра» — домашнюю командную игру-викторину,
работающую в локальной сети (LAN): один сервер, игроки на телефонах, табло на ТВ/проекторе,
ведущий на ноутбуке.

ИСТОЧНИК ИСТИНЫ ПО ДИЗАЙНУ
Папка `design_handoff_svoya_igra/`:
- `README.md` — полная спецификация: экраны, состояния, игровой цикл, модель состояния,
  дизайн-токены (цвета/шрифты/радиусы/отступы/тени/движение), доступность.
- `Своя игра - Студия.dc.html` — рабочий дизайн-прототип (high-fidelity). Открой в браузере
  и пройди по навигатору слева, чтобы увидеть все экраны и состояния вживую.
- `screenshots/` — статичные снимки ключевых экранов.
Прежде чем писать код, прочитай README.md целиком и изучи прототип.

ВАЖНО ПРО ПРОТОТИП
`.dc.html` — это дизайн-референс, НЕ продакшн-код. В нём встроен рантайм `support.js`
(только для просмотра) — не тащи его в продукт. Команды-соперники там «нажимают» buzzer на
таймерах для демонстрации; состояние игры живёт в одном клиенте. В реальном продукте
нажатия приходят с устройств по сети, а состояние комнаты живёт на сервере и
синхронизируется всем участникам.

СТЕК (поправь под свой проект)
- Сервер реального времени: Node + WebSocket (или Socket.IO). Авторитетное состояние
  комнаты на сервере; клиенты получают снапшоты/события и шлют действия.
- Клиент: React + TypeScript + Vite. Роутинг по экранам (см. пути в README).
- Состояние комнаты — конечный автомат фаз (см. ниже). Без БД для MVP достаточно памяти
  процесса; контент базы вопросов — JSON-файл с импортом/экспортом.

ЧТО ПОСТРОИТЬ (приоритет сверху вниз)
1. Сервер-комната + протокол: подключение игрока/табло/ведущего, список команд/игроков,
   онлайн/офлайн, синхронизация состояния.
2. Игровой цикл (автомат фаз): select → question → buzzing → answering → judged → next.
   - Очередь buzzer с дельтами времени (сортировка по времени нажатия).
   - Судейство Верно/Неверно (±цена; при «Неверно» ход следующему в очереди, иначе показ
     ответа).
   - Таймер ответа 30–60 с (настройка ведущего): старт при входе в answering; на нуле —
     не засчитано (как «Неверно»), ход следующему; перезапуск для каждой команды.
   - Фальстарт: нажатие до открытия buzzer → блок ~3 с (НЕ новый круг). На клиенте игрока:
     кнопка краснеет, отсчёт кольцом по ободу, текст «Фальстарт!».
3. Экраны (точно по README §5 и токенам §8):
   Лендинг, Вход игрока, Экран игрока (buzzer + все состояния), Табло (сетка + оверлей
   вопроса + очередь + счёт + итоги/победитель), Лобби, Пульт ведущего, Конструктор игры
   (drag&drop + лайв-валидация + публикация), База вопросов (типы текст/картинка/аудио +
   автосохранение).
4. Мини-игры «Кот в мешке» (передача вопроса) и «Аукцион» (ставки) — контролы у ведущего.

ТРЕБОВАНИЯ К КАЧЕСТВУ
- Язык интерфейса — русский, на «вы», дружелюбный, без канцелярита; кнопки — активным
  глаголом; имя действия сохраняется через флоу («Опубликовать» → «Опубликовано»).
- Три вьюпорта: телефон (одна рука, крупный buzzer ≥210px), ТВ (читаемость с 2–5 м, крупная
  типографика, высокий контраст), ноутбук (плотные инструменты). Адаптивность обязательна.
- Доступность: видимый фокус с клавиатуры; `prefers-reduced-motion` отключает анимации;
  крупные цели нажатия.
- Тёмное окружение: не слепить и не тонуть. Токены строго из README §8 (палитра, Oswald +
  Manrope, радиусы, тени/свечения, анимации popIn/pulseRing/spin/кольцо).
- Инвариант: текст вопроса + ОТВЕТ видит только ведущий; на табло и у игрока ответ не
  показывается (кроме момента раскрытия правильного ответа после раунда вопроса).

ПОРЯДОК РАБОТЫ
Сначала предложи короткий план и структуру файлов, подтверди стек, затем реализуй
вертикальными срезами: (а) сервер-комната + подключение, (б) полный цикл одного вопроса с
buzzer/таймером/судейством на трёх экранах, (в) остальные экраны, (г) конструктор и база.
После каждого среза — кратко что готово и как проверить.
```
