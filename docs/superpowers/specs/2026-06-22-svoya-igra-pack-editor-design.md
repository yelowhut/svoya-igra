# Редактор паков «Своя игра» — дизайн (Sub-project 2)

Дата: 2026-06-22
Статус: согласован, готов к написанию плана

## Цель

UI для составления игры (раунды → категории → вопросы, включая медиа и
бонусные типы) с серверным хранением черновиков, прямой публикацией в играбельный
пак, опциональным экспортом в ZIP того же формата и тестовым прогоном через готовый
движок.

Контекст: движок (Sub-project 1) завершён. Формат пака — `docs/pack-format.md`.
Strict-схема пака — `src/packs/schema.ts` (`gameJsonSchema`), импорт ZIP —
`src/packs/import.ts`, HTTP — `src/http/server.ts`, веб — Svelte+Vite с входами
`index.html`/`play.html`/`board.html`.

## Ключевые решения

1. **Серверное хранение черновиков.** Отдельная таблица `drafts`, отдельные REST-эндпоинты.
2. **Прямая публикация + опциональный ZIP.** «Опубликовать» пишет в `packs`; ZIP-экспорт
   и ZIP-импорт — для бэкапа/переноса.
3. **Медиа грузятся сразу на сервер** (папка черновика), в черновике хранится только путь.
4. **«Локальный тест» = полный solo-прогон** через существующий движок.
5. **Умная публикация.** Перед перезаписью проверяем ссылающиеся игры, предупреждаем,
   даём выбор «новый пак» / «перезаписать».

## Архитектура

### Размещение и сборка

Редактор — **четвёртый Vite-вход** в существующем `web/`:

- `web/editor.html` + `web/src/editor/` (`main.ts`, `App.svelte`, компоненты).
- Добавить вход `editor` в `web/vite.config.ts` (`rollupOptions.input`).
- Сервер отдаёт чистый роут `GET /editor` → `editor.html` (по образцу `/host`,
  `/play`, `/board` в `src/http/server.ts`).

Отдельного приложения нет — переиспользуем сборку, тему (`web/src/lib/theme.css`),
сокет/identity не нужны (редактор работает по REST).

### Модель данных: две схемы

Черновик в процессе редактирования нарушает strict-схему (пустые названия, ещё не
введённые ответы, пустые массивы). Поэтому **две схемы**:

- **Relaxed (черновик)** — новый файл `src/packs/draftSchema.ts`:
  - все строки допускают пустое значение; массивы могут быть пустыми;
  - у каждого раунда/категории/вопроса — **стабильный `id`** (string), сгенерированный
    при создании элемента и **сохраняемый между сессиями** (не регенерируется при
    загрузке);
  - `value` — целое ≥ 0; `type` ∈ {text,image,audio}; `special` ∈ {none,auction,cat};
  - `media` — опциональный путь (`media/<file>` относительно папки черновика);
  - дополнительное поле верхнего уровня `lastPublishedPackId?: string`.
- **Strict (публикация)** = существующая `gameJsonSchema` (`min(1)` на title/name/
  prompt/answer, непустые массивы, media обязателен для image/audio).

Лайв-валидация прогоняет strict-проверку поверх relaxed-данных и возвращает список
проблем (путь к полю + сообщение), которые UI подсвечивает. Strict-валидация — также
gate публикации.

Хранимый формат черновика **отличается** от `game.json`: содержит стабильные id и
`lastPublishedPackId`. Преобразование draft→`game.json` (отбрасывает id, оставляет
контент) используется при публикации, экспорте и валидации.

### Хранение и серверный API

Новая таблица (миграция в `src/persistence/db.ts`):

```sql
CREATE TABLE IF NOT EXISTS drafts (
  id         TEXT PRIMARY KEY,
  data       TEXT NOT NULL,        -- JSON relaxed-черновика
  updated_at INTEGER NOT NULL
);
```

Эндпоинты (новый модуль роутов, регистрируется в `buildServer`; без auth — как весь
движок на доверенной LAN):

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/drafts` | список `{id, title, updatedAt}` |
| POST | `/api/drafts` | создать черновик (пустой или из шаблона по телу `{template?}`) → `{id}` |
| GET | `/api/drafts/:id` | получить relaxed-черновик целиком |
| PUT | `/api/drafts/:id` | сохранить (вызывается авто-сохранением, debounce ~400 мс на клиенте) |
| DELETE | `/api/drafts/:id` | удалить черновик + его папку медиа |
| POST | `/api/drafts/:id/media` | загрузка файла (multipart) → сохранить в папку черновика, вернуть `{path}` |
| POST | `/api/drafts/:id/publish` | strict-валидация → запись в `packs`; тело `{mode}` |
| GET | `/api/drafts/:id/publish/preflight` | вернуть `{published: bool, referencingGames: number}` для предупреждения UI |
| POST | `/api/drafts/import` | приём ZIP (multipart) → новый черновик → `{id}` |
| GET | `/api/drafts/:id/export` | собрать и отдать ZIP (`game.json` + media) |

`POST /api/drafts/:id/publish` тело: `{ mode: "new" | "overwrite" }`.
- `new` → новый `packId` (`crypto.randomUUID()`), `INSERT` в `packs`, медиа копируются
  в папку нового пака, `lastPublishedPackId` черновика обновляется.
- `overwrite` → использовать `lastPublishedPackId` как `packId`, `INSERT OR REPLACE`,
  перезаписать медиа пака. Допустимо только если `lastPublishedPackId` задан.

Ответ публикации: `{ packId, title, rounds }` (как у текущего `POST /api/packs`).

Эндпоинты `packs`/`games` остаются без изменений; черновики **не** идут через
`/api/packs`.

### Жизненный цикл медиа

- Папка черновика: `{mediaDir}/drafts/{draftId}/media/...`.
- Загрузка (`POST /api/drafts/:id/media`): лимит размера (напр. 25 МБ, через опции
  `@fastify/multipart`), белый список MIME — `image/jpeg`, `image/png`, `image/webp`,
  `audio/mpeg`, `audio/ogg`, `audio/mp4`. Имя файла санитизируется; защита от path
  traversal (по образцу `import.ts`). Превью в UI — через существующий
  `GET /media/...`? Нет: медиа черновика лежит вне `{mediaDir}/{packId}`, поэтому
  добавить отдачу медиа черновика (роут `GET /media/drafts/:id/*` или расширить
  существующий обработчик с проверкой пути).
- При публикации: файлы **копируются** (не перемещаются) в `{mediaDir}/{packId}/media/...`,
  черновик остаётся пригодным для дальнейшей правки.
- GC осиротевших файлов: при `PUT /api/drafts/:id` сравнить ссылки `media` в JSON с
  файлами на диске папки черновика и удалить неиспользуемые.

### UI редактора

Стартовый экран — **список черновиков**: создать пустой / создать из шаблона / импорт
ZIP / открыть / удалить.

Экран редактирования пака:

- Заголовок игры (поле `title`).
- Вкладки/список **раундов**; внутри раунда — **гибкая сетка категории × тиры**:
  - по умолчанию шаблон **5×5** с авто-тирами `100..500 × (номер раунда)`;
  - категории и вопросы добавляются/убираются свободно (6×5, 5×6 и т.п.) — сетка не
    обязана быть прямоугольной.
- **Карточка вопроса**: `prompt`, `answer`, `value`, `type` (text/image/audio),
  загрузка медиа с превью (img/audio), `special` (none/auction/cat).
- **Операции**: добавить/удалить раунд/категорию/вопрос; переупорядочить (кнопки
  вверх/вниз); дублировать категорию/вопрос (с генерацией новых id).
- **Лайв-валидация**: подсветка пустых/некорректных полей по ходу правки (strict
  поверх relaxed).
- **Панель действий**: индикатор авто-сохранения, «Скачать ZIP», «Опубликовать»,
  «Сыграть тестовую».

Состояние черновика на клиенте — Svelte store; мутации помечают «грязное» состояние
и триггерят debounced `PUT`.

### Публикация и тест

- **«Опубликовать»**: запустить клиентскую strict-валидацию; при ошибках —
  заблокировать и показать список. Иначе вызвать `preflight`; если
  `published && referencingGames > 0` — модалка с выбором *новый пак* / *перезаписать*
  (по умолчанию — новый, как безопасный). Если черновик ещё не публиковался — сразу
  `mode:"new"`.
- **«Сыграть тестовую»**: publish (`mode:"new"`) → `POST /api/games` с title пака и
  дефолтным `teamCount` → открыть `/host?game=<id>` и `/board?game=<id>` (новые
  вкладки). Полностью переиспользует готовый движок.

## Тестирование

Серверная сторона (основной фокус, в `src/http/` рядом с `server.test.ts` либо новый
файл):

- CRUD черновиков (create/list/get/update/delete).
- Media upload: успешная загрузка, отклонение неверного MIME/размера, защита от
  path traversal, GC осиротевших файлов при `PUT`.
- Публикация: `mode:"new"` создаёт новый пак и копирует медиа; `mode:"overwrite"`
  перезаписывает; `preflight` корректно считает ссылающиеся игры; strict-валидация
  блокирует невалидный черновик.
- Import/export round-trip: ZIP → черновик → ZIP сохраняет контент.
- Relaxed/strict схемы: relaxed принимает неполный черновик; draft→game.json + strict
  ловит пропуски.

Клиент: 1–2 smoke-теста на критичные утилиты (валидатор draft→strict; преобразование
draft-state — добавление/дублирование/переупорядочивание сохраняет стабильные id).

## Вне области (MVP)

- **«Финал»** (тайные ставки) — заложен в формат пака, но движок его не играет;
  редактор его не поддерживает.
- **Аутентификация** — отсутствует во всём движке (backlog `svoya-igra-status`);
  эндпоинты редактора тоже без auth, консистентно. Хардить при выносе наружу.
- **Drag&drop** переупорядочивания — в MVP только кнопки вверх/вниз.
