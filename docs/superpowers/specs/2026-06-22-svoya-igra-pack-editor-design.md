# Редактор паков «Своя игра» — дизайн (Sub-project 2)

Дата: 2026-06-22
Статус: согласован, готов к написанию плана

## Цель

Инструмент для подготовки игр: отдельно **управление базой вопросов** (контент) и
отдельно **сборка игры** (сетка, цены, раскладка вопросов, мини-игры). Игра
публикуется в существующий формат пака и играется готовым движком без изменений.

Контекст: движок (Sub-project 1) завершён. Текущий формат пака — `docs/pack-format.md`
(flat: rounds → categories → questions с inline `value`/`special`). Strict-схема —
`src/packs/schema.ts`, импорт ZIP — `src/packs/import.ts`, HTTP — `src/http/server.ts`,
веб — Svelte+Vite (входы `index.html`/`play.html`/`board.html`).

## Ключевая идея: цена — свойство сетки, а не вопроса

Раунд игры — **прямоугольная сетка**: строки-категории × столбцы-цены. Цена задаётся
**столбцу** и одинакова для всех категорий этого столбца (`[кат1; цена5]` стоит столько
же, сколько `[кат2; цена5]`). Вопрос **не несёт** ни цену, ни тип мини-игры — он лишь
кладётся в ячейку и при публикации получает `value` своего столбца и `special` тега
ячейки. Поэтому контент (вопрос/ответ/медиа) и игровая компоновка (цены, размещение,
мини-игры) — **разные сущности**.

## Две подсистемы

- **2a — Базы вопросов** (шаблоны ВОПРОСОВ): единый общий пул. Плоский список
  **категорий**, в каждой — **вопросы** (тип/текст/ответ/медиа). Без цен, без сетки, без
  мини-игр. Это фундамент и строится первым.
- **2b — Редактор игр** (шаблоны ИГР): сетки с ценами, раскладка вопросов из банка по
  ячейкам (drag&drop), распределение сеток по раундам, назначение мини-игр на ячейки,
  публикация и тест.

2b ссылается на 2a, поэтому реализуется после. См. «Стадирование».

## Ключевые решения

1. **Серверное хранение** обеих подсистем (без auth — как весь движок на доверенной LAN).
2. **Стабильные UID** (`crypto.randomUUID()`) на категории и вопросы банка. UID —
   одновременно ключ ссылки из ячейки игры и ключ дедупликации при экспорте/импорте банка.
3. **Ячейка игры ссылается на вопрос банка по UID** (живая связь): правка вопроса в банке
   видна во всех играх до публикации; при публикации делается снимок (flatten).
4. **Три раздельные операции переноса данных** (см. «Экспорт/импорт/публикация»):
   экспорт/импорт банка, экспорт/импорт шаблона игры, публикация игры в движок.
5. **Прямоугольная сетка**; **мини-игра — тег на ячейке** (none/auction/cat,
   взаимоисключающе).
6. **Drag&drop** — основной способ размещения вопросов в сетке (нативный HTML5 DnD,
   без внешних библиотек). Прочее переупорядочивание — кнопки вверх/вниз.

## Размещение и сборка (web)

## Модель данных

### Банк (2a) — нормализованные таблицы

Нормализованные таблицы (а не JSON-blob) — чтобы preflight-запросы «кто ссылается на этот
вопрос» и гранулярные правки были дешёвыми. Миграции в `src/persistence/db.ts`:

```sql
CREATE TABLE IF NOT EXISTS bank_categories (
  id       TEXT PRIMARY KEY,   -- UID
  name     TEXT NOT NULL,
  position INTEGER NOT NULL
);
CREATE TABLE IF NOT EXISTS bank_questions (
  id          TEXT PRIMARY KEY,            -- UID (ключ ссылки + дедупликации)
  category_id TEXT NOT NULL REFERENCES bank_categories(id),
  type        TEXT NOT NULL,               -- text | image | audio
  prompt      TEXT NOT NULL DEFAULT '',
  answer      TEXT NOT NULL DEFAULT '',
  media       TEXT,                        -- путь bank/media/<file>, для image/audio
  position    INTEGER NOT NULL
);
```

Вопрос банка **не** имеет `value` и `special` — они появляются только в игре.

### Игра (2b) — JSON-документ

Таблица `game_templates (id TEXT PK, data TEXT, updated_at INTEGER)` (JSON-blob +
debounced PUT). Документ:

```ts
{
  id, title,
  lastPublishedPackId?: string,
  rounds: [{
    id, name,
    columns: [{ id, value }],            // N цен-столбцов, по возрастанию
    rows:    [{
      id,
      categoryId,                        // привязка к bank_categories.id
      cells: [{ columnId, questionId?, special }]  // special: none|auction|cat
    }]
  }]
}
```

Сетка прямоугольная: у каждой строки ровно те же столбцы, что у раунда. В процессе
редактирования ячейки могут быть пустыми (`questionId` отсутствует); публикация требует
все заполненными. Имя выходной категории = имя банк-категории на момент публикации
(override на строке в MVP **не** делаем).

### Strict-схема публикации (новый слой)

Перед публикацией проверяется (не существующая `gameJsonSchema`, а слой раньше):

- `title` непуст; ≥1 раунд; у раунда непустое `name`, ≥1 столбец, ≥1 строка;
- каждый `column.value` — целое > 0;
- каждый `row.categoryId` существует в банке;
- каждая ячейка имеет `questionId`; вопрос существует в банке и **принадлежит**
  `row.categoryId`;
- для вопросов `image`/`audio` присутствует `media`.

Лайв-валидация прогоняет те же правила по ходу редактирования и подсвечивает проблемные
ячейки/строки/поля.

## Серверный API

Без auth. Префиксы разведены, чтобы не конфликтовать с существующими `/api/packs`
(импорт пака движком) и `/api/games` (создание игровых инстансов движка).

### Банк — `/api/bank`

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/bank/categories` | список категорий (id, name, position, кол-во вопросов) |
| POST | `/api/bank/categories` | создать категорию → `{id}` |
| PUT | `/api/bank/categories/:id` | переименовать / переместить (position) |
| DELETE | `/api/bank/categories/:id` | удалить (с preflight, см. целостность) |
| GET | `/api/bank/categories/:id/questions` | вопросы категории |
| POST | `/api/bank/categories/:id/questions` | создать вопрос → `{id}` |
| PUT | `/api/bank/questions/:id` | правка (prompt/answer/type/media/position) |
| DELETE | `/api/bank/questions/:id` | удалить (с preflight) |
| POST | `/api/bank/questions/:id/media` | загрузка медиа (multipart) → `{path}` |
| GET | `/api/bank/export` | `bank.zip` (bank.json + media) |
| POST | `/api/bank/import` | импорт `bank.zip`, слияние по UID |

### Шаблоны игр — `/api/game-templates`

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/game-templates` | список `{id, title, updatedAt}` |
| POST | `/api/game-templates` | создать (пустой / из шаблона `{template?}`) → `{id}` |
| GET | `/api/game-templates/:id` | получить документ |
| PUT | `/api/game-templates/:id` | сохранить (auto-save, debounce ~400 мс) |
| DELETE | `/api/game-templates/:id` | удалить |
| GET | `/api/game-templates/:id/publish/preflight` | `{published, referencingGames}` |
| POST | `/api/game-templates/:id/publish` | strict-валидация → flatten → запись в `packs`; тело `{mode}` |
| GET | `/api/game-templates/:id/export` | экспорт шаблона (сетка + ссылки по UID) |
| POST | `/api/game-templates/import` | импорт шаблона → новый документ |

`publish` тело `{ mode: "new" | "overwrite" }`:
- `new` → новый `packId`, `INSERT` в `packs`, медиа копируются в папку пака,
  `lastPublishedPackId` обновляется;
- `overwrite` → использовать `lastPublishedPackId` как `packId`, `INSERT OR REPLACE`;
  допустимо только если он задан.

UI вызывает `preflight`; если `published && referencingGames > 0` — модалка с выбором
«новый пак» / «перезаписать» (по умолчанию — новый, безопасный).

## Медиа-цикл

- **Банк**: файлы в `{mediaDir}/bank/media/<file>` (имя санитизируется и делается
  уникальным, напр. префикс UID вопроса). Загрузка: лимит размера (~25 МБ через опции
  `@fastify/multipart`), белый список MIME — `image/jpeg|png|webp`, `audio/mpeg|ogg|mp4`;
  защита от path traversal (по образцу `import.ts`).
- **Отдача для превью**: новый роут отдачи банк-медиа (или расширение текущего
  `/media/...` с проверкой пути).
- **GC**: при удалении вопроса с media; при `PUT`, если поле `media` обнулилось/сменилось.
- **На публикацию**: файлы **копируются** (не move) в `{mediaDir}/{packId}/media/...`,
  пути в паке переписываются `bank/media/<file>` → `media/<file>`.

## Экспорт / импорт / публикация (три раздельные ветки)

1. **Экспорт/импорт банка** — `bank.zip` (`bank.json` с UID + `media/`). Импорт сливает
   по UID: совпал UID → обновить, нет → добавить. Идемпотентно; переносит банк между
   машинами с сохранением идентичности (ссылки шаблонов остаются валидны).
2. **Экспорт/импорт шаблона игры** — сетка + цены + теги + ссылки на вопросы по UID, без
   медиа (оно в банке). Импорт резолвит ссылки против текущего банка; отсутствующие UID →
   предупреждение, ячейка помечается невалидной (требует пере-привязки).
3. **Публикация в движок** — односторонний flatten в текущий flat-формат пака (см. ниже)
   + копирование медиа → запись в `packs`. Не обратима в редактор.

### Flatten-правила (публикация)

Для каждого раунда → для каждой строки → выходная категория:
- `name` = имя банк-категории на момент публикации (snapshot);
- `questions` — по возрастанию `column.value`; каждый: контент резолвится из банк-вопроса
  (`type`/`prompt`/`answer`/`media`), `value` = цена столбца, `special` = тег ячейки;
- media-пути переписываются и файлы копируются (см. медиа-цикл).

Результат = существующий формат пака → проходит существующую `gameJsonSchema`. Движок не
меняется.

## UI

### 2a — Менеджер базы вопросов (фундамент, делаем первым)

Это собственно **создание контента**. Экран:

- Слева — **дерево/список категорий** (создать, переименовать, удалить, вверх/вниз).
- Справа — **вопросы выбранной категории** (карточки), создать/править/удалить, вверх/вниз.
- **Форма вопроса**: `type` (text/image/audio), `prompt`, `answer`; для image/audio —
  загрузка медиа с превью (img/audio-плеер). Без полей цены/мини-игры.
- Кнопки экспорта/импорта банка.
- Авто-сохранение через гранулярные endpoints (правка одного вопроса = один PUT).

### 2b — Редактор игр

- Список игр (создать пустую / из шаблона 5×5 / импорт шаблона / открыть / удалить).
- Экран игры: `title` + раунды (вкладки).
- Раунд — **прямоугольная сетка**: заголовки-столбцы с **редактируемой ценой**; строки —
  категории. По умолчанию шаблон 5×5, авто-цены `100..500 × номер раунда`; столбцы и
  строки добавляются/убираются (остаётся прямоугольной).
- **Боковая панель**: дерево банк-категорий (мини-карточки). Перетаскиваешь **категорию**
  на пустую строку → строка привязывается (`categoryId`), список вопросов фильтруется по
  этой категории (мини-превью). Перетаскиваешь **вопросы** на ячейки — **только в строку
  своей категории** (в чужой ряд нельзя).
- **Ячейка**: превью назначенного вопроса + переключатель тега мини-игры (кот / весы-
  аукцион / нет, взаимоисключающе).
- **Лайв-валидация**: подсветка пустых/невалидных ячеек/строк.
- **Панель действий**: индикатор авто-сохранения, «Скачать шаблон», «Опубликовать»,
  «Сыграть тестовую».
- «Сыграть тестовую»: publish (`mode:"new"`) → `POST /api/games` (title пака,
  дефолтный `teamCount`) → открыть `/host?game=<id>` и `/board?game=<id>`.

## Корнер-кейсы целостности

- **Удаление вопроса из банка**: preflight → список игр с этим UID; при подтверждении в
  играх ячейки очищаются (`questionId=null`) и помечаются невалидными.
- **Удаление категории из банка**: preflight на привязки строк; при подтверждении строки
  игр остаются с висящей `categoryId` (binding broken) — вся строка инвалидна до
  пере-привязки.
- **Перенос вопроса между категориями банка**: **запрещён** (только delete+создать заново)
  — иначе ячейка нарушит инвариант «вопрос принадлежит категории строки».
- **Смена `categoryId` у строки игры**: все ячейки строки **очищаются** (с подтверждением).

## Тестирование

Сервер (основной фокус, новые файлы в `src/http/` / `src/packs/`):

- Банк: CRUD категорий/вопросов; media upload (успех, отклонение MIME/размера, path
  traversal, GC); экспорт/импорт банка (слияние по UID идемпотентно).
- Игры: CRUD; preflight (счёт ссылающихся игр); публикация `new`/`overwrite`;
  strict-валидация нового слоя; flatten корректно мапит цену столбца и тег ячейки,
  порядок вопросов, snapshot имени категории, копирование медиа; экспорт/импорт шаблона
  (резолв ссылок, предупреждение об отсутствующих UID).
- Целостность: удаление вопроса/категории очищает/инвалидирует ячейки; смена привязки
  строки очищает ячейки; запрет переноса вопроса между категориями.

Клиент: 1–2 smoke-теста на критичные утилиты (валидатор игры→strict; draft-state grid:
добавление/дублирование/переупорядочивание сохраняет стабильные id; ограничение DnD
«вопрос только в свою строку»).

## Стадирование реализации

Один дизайн, два плана (связь по ссылкам центральна — держим в одном spec):

- **2a — Базы вопросов**: таблицы банка, `/api/bank/*` CRUD + медиа + экспорт/импорт,
  UI менеджера контента. Полезна сама по себе.
- **2b — Редактор игр**: `game_templates`, сетка/цены, drag&drop размещение, мини-игры,
  публикация+flatten, экспорт/импорт шаблона, solo-тест, корнер-кейсы целостности.

2b строится на 2a.

## Вне области (MVP)

- **ZIP-импорт чужого пака движка обратно в редактор** — не делаем (плохо ложится на
  ссылочную модель; перенос между установками — через экспорт/импорт банка и шаблона).
- **«Финал»** (тайные ставки) — в формате, но движок не играет; редактор не поддерживает.
- **Аутентификация** — отсутствует во всём движке (backlog `svoya-igra-status`),
  консистентно. Хардить при выносе наружу.
- **Override имени категории** на строке игры; **перенос вопроса между категориями**;
  **drag&drop** для переупорядочивания раундов/столбцов/категорий (только кнопки).
