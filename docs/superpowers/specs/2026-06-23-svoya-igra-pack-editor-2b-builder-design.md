## Цель

Заполнить заглушку `web/src/admin/sections/Builder.svelte` полноценным конструктором игр:
сборка игры из вопросов банка (сетка строки-категории × столбцы-цены, drag&drop
размещение, теги мини-игр), публикация в существующий формат пака и тест готовым движком
без его изменений.

Контекст: 2-shell (админ-оболочка `/admin`, cookie-auth, левый рейл «Студия», клиентский
роутер) и 2a (банк вопросов — таблицы `bank_categories`/`bank_questions`, `/api/bank/*`,
UI базы вопросов) реализованы и влиты в `main`. Движок (SP1) завершён; текущий формат
пака — `docs/pack-format.md`; strict-схема пака — `src/packs/schema.ts` (`gameJsonSchema`).

## Ключевая идея (из umbrella, неизменна)

Раунд — **прямоугольная сетка** строки-категории × столбцы-цены. Цена — свойство
**столбца**, одинакова для всех категорий столбца. Вопрос **не несёт** ни цену, ни тип
мини-игры; он лишь кладётся в ячейку и при публикации получает `value` своего столбца и
`special` тега ячейки. Контент (банк, 2a) и игровая компоновка (этот документ) — разные
сущности. Ячейка ссылается на вопрос банка **по UID** (живая связь): правка вопроса в
банке видна во всех играх до публикации; при публикации делается снимок (flatten).

## Объём и стадирование

Этот spec покрывает обе под-стадии; план writing-plans начинается с core.

- **2b-core** — модель + CRUD `game_templates`; экран «Список игр»; сетка-редактор с
  ценами; DnD категорий и вопросов; теги мини-игр; лайв-валидация; публикация + flatten;
  «Сыграть тестовую». Это минимально играбельный поток «собрал → опубликовал → сыграл».
- **2b-ext** — экспорт/импорт шаблона игры (сетка + цены + теги + ссылки на вопросы по
  UID, без медиа); кнопка «Скачать шаблон» и импорт в «Список игр».

## Решения brainstorming-сессии (отличия от umbrella)

1. **Целостность ссылок — мягкая инвалидация (не preflight в банке).** Банк не трогаем:
   cascade-delete 2a остаётся как есть, **ноль правок `src/persistence/bankRepo.ts`**.
   Конструктор при загрузке/редактировании резолвит `questionId`/`categoryId` против
   банка и помечает висящие ячейки/строки невалидными через лайв-валидацию. Корнер-кейс
   umbrella «удаление вопроса/категории → preflight-сканирование игр» **снят** — заменён
   инвалидацией на стороне конструктора. Это робастнее (любая рассинхронизация ловится при
   открытии шаблона) и не плодит cross-cut правки во влитую 2a.
2. **`publish/preflight.referencingGames` = живые игровые инстансы движка** по
   `lastPublishedPackId`. Игры хранятся event-sourcing (таблица `events`,
   `type='GAME_CREATED'`, `payload.packId`) — считаем запросом по `events`, **новых таблиц
   не нужно**. Перезапись пака под играющей игрой → активные игры **принудительно
   завершаются** (см. «Корректировки 2026-06-23», п.1; это переопределяет «безопасный
   новый пак» из umbrella).
3. **DnD-консистентность с 2a.** Переиспользуем паттерны, отлаженные в базе вопросов:
   **линия-индикатор вставки** при сортировке (а не рамка вокруг наводимого), `Modal.svelte`
   для подтверждений (вместо `prompt`/`confirm` где уместно), inline-SVG иконки
   (грип/мусорка/карандаш). Нативный HTML5 DnD, без внешних библиотек. 2a не вынес DnD в
   общий модуль (логика внутри `CategoryList`/`QuestionList`) — для 2b при необходимости
   выделим лёгкий хелпер.
4. **`PRAGMA foreign_keys=ON` — в backlog, не в 2b.** Целостность ссылок держим
   лайв-валидацией по UID на стороне конструктора, не на уровне БД.

Подтверждённые инварианты 2a: стабильные UUID как ключ ссылки; запрет смены `category_id`
вопроса; медиа-модель flatten `bank/media/<f>` → `media/<f>`.

## Корректировки brainstorming-сессии 2026-06-23 (итерация 2)

> Эти решения уточняют разделы ниже после анализа спека «сыграем на выходных». **При
> расхождении с разделами ниже приоритет здесь.**

1. **Перезапись пака → принудительное завершение активных игр (упрощение).** Принимаем
   как данность: при публикации в существующий packId все активные игровые инстансы на
   этом паке **принудительно завершаются**. Контекст — играем сами, больше одной активной
   комнаты на пак это редчайший кейс, на параллельные комнаты не закладываемся. Это
   снимает класс проблем «перезапись под живой игрой ломает состояние» (движок читает пак
   из `packs` на каждый `broadcastState` через `loadPack`, а `parseGameJson` генерирует
   новые UUID → старые `usedQuestionIds`/`currentQuestionId` в event-sourced state теряют
   ссылки). Реализация завершения: для каждой игры на этом packId дописать в `events`
   форс-событие конца игры (`GAME_END`), чтобы доска/ведущий перешли в финальный экран и
   не читали уже перезаписанный контент. Модалка публикации остаётся, но как простое
   предупреждение: «N активных игр на этом паке будут завершены. Продолжить?».
   `preflight.referencingGames` меняет смысл — «сколько игр будет закрыто», а не «опасно».

2. **«Сыграть тестовую» переиспользует один packId черновика.** В документе шаблона
   держим `lastPublishedPackId`; «Сыграть тестовую» публикует в **тот же** packId
   (с принудительным завершением прошлой тестовой игры по п.1). Так база и `mediaDir` не
   захламляются десятками мёртвых тестовых паков за вечер сборки. `teamCount` тестовой
   игры — дефолт **3** (это meta/max, команды ведущий заводит уже в игре).

3. **Проверка медиа — физическое наличие файла.** Strict-валидация публикации для
   `image`/`audio`-вопроса проверяет не только заданность пути `media`, но и что файл
   **реально существует** в `bank/media/`. Нет файла → публикация блокируется, ячейка
   подсвечена. Лайв-валидация помечает ячейку с битым медиа невалидной (как висящий UID).
   В превью ячейки — **мини-плеер аудио / тумба картинки**, чтобы проверять медиа без
   запуска «Сыграть тестовую».

4. **Цена столбца.** Авто-цена `100..N00 × номер_раунда` распространяется на 6-й+ столбец
   (6-й = `600 × раунд` и т.д.). Дополнительно — **ручной ввод** цены столбца, **кратно
   100**, целое > 0. Дубль одинаковых цен в раунде (две «200») — **мягкое предупреждение**
   в лайв-валидации, не блок.

5. **Пробелы strict-валидации.** (а) Дубль одного `questionId` в нескольких ячейках —
   **предупреждение**, не блок (на игре вопрос выскочит дважды). (б) Тег мини-игры
   (Кот/Аукцион) можно ставить **только на ячейку с назначенным вопросом**; очистка ячейки
   сбрасывает тег. (в) Снапшот при публикации касается не только имени категории — **`title`
   игры и `name` раунда** тоже фиксируются на момент публикации (берутся из самого
   документа шаблона).

6. **Flush авто-сейва перед действием.** Перед «Опубликовать» и «Сыграть тестовую» —
   принудительный flush debounced-PUT, иначе из-за гонки debounce (~400 мс) опубликуется
   предыдущий черновик.

7. **Финал — отдельный спек 2c.** Финальный раунд (тайные ставки, вычёркивание тем,
   тайные ответы, верификация ведущим) — полноценная вертикальная фича через формат пака,
   движок, UI host/board/player и конструктор. Выносится в самостоятельный спек **2c
   «Финальный раунд»**, реализуется после 2b-core. В 2b конструктор финал **не строит**.

## Модель данных (из umbrella, неизменна)

Таблица (миграция в `src/persistence/db.ts`):

```sql
CREATE TABLE IF NOT EXISTS game_templates (
  id         TEXT PRIMARY KEY,
  data       TEXT NOT NULL,      -- JSON-документ
  updated_at INTEGER NOT NULL
);
```

JSON-документ (debounced PUT):

```ts
{
  id, title,
  lastPublishedPackId?: string,
  rounds: [{
    id, name,
    columns: [{ id, value }],            // N цен-столбцов, по возрастанию
    rows:    [{
      id,
      categoryId,                        // привязка к bank_categories.id
      cells: [{ columnId, questionId?, special }]  // special: none|auction|cat
    }]
  }]
}
```

Сетка прямоугольная: у каждой строки ровно те же столбцы, что у раунда. В процессе
редактирования ячейки могут быть пустыми (`questionId` отсутствует); публикация требует
все заполненными. Имя выходной категории = имя банк-категории на момент публикации
(override на строке в MVP **не** делаем).

## Серверный API — `/api/game-templates/*`

Все роуты за `requireAdmin` (preHandler из 2-shell). Префикс разведён с существующими
`/api/packs` и `/api/games`.

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/game-templates` | список `{id, title, updatedAt}` |
| POST | `/api/game-templates` | создать (пустой / из шаблона 5×5 `{template?}`) → `{id}` |
| GET | `/api/game-templates/:id` | получить документ |
| PUT | `/api/game-templates/:id` | сохранить (auto-save, debounce ~400 мс) |
| DELETE | `/api/game-templates/:id` | удалить |
| GET | `/api/game-templates/:id/publish/preflight` | `{published, referencingGames}` |
| POST | `/api/game-templates/:id/publish` | strict-валидация → flatten → запись в `packs`; тело `{mode}` |
| GET | `/api/game-templates/:id/export` | экспорт шаблона (сетка + ссылки по UID) — **2b-ext** |
| POST | `/api/game-templates/import` | импорт шаблона → новый документ — **2b-ext** |

### preflight (уточнение сессии)

Возвращает `{ published: boolean, referencingGames: number }`:
- `published` = задан ли `lastPublishedPackId`;
- `referencingGames` = число живых игровых инстансов движка по этому packId. Считается
  запросом по `events` (`type='GAME_CREATED'`, парсинг `payload.packId`).

UI: если `referencingGames > 0` → модалка-предупреждение «N активных игр на этом паке
будут завершены. Продолжить?» (см. «Корректировки 2026-06-23», п.1).

### publish тело `{ mode: "new" | "overwrite" }`

- `new` → новый `packId`, `INSERT` в `packs`, медиа копируются в папку пака,
  `lastPublishedPackId` обновляется;
- `overwrite` → использовать `lastPublishedPackId` как `packId`, `INSERT OR REPLACE`;
  допустимо только если он задан.

## Strict-схема публикации (новый слой)

Перед публикацией проверяется (не существующая `gameJsonSchema`, а слой раньше):

- `title` непуст; ≥1 раунд; у раунда непустое `name`, ≥1 столбец, ≥1 строка;
- каждый `column.value` — целое > 0;
- каждый `row.categoryId` существует в банке;
- каждая ячейка имеет `questionId`; вопрос существует в банке и **принадлежит**
  `row.categoryId`;
- для вопросов `image`/`audio` присутствует `media`.

Лайв-валидация прогоняет те же правила по ходу редактирования и подсвечивает проблемные
ячейки/строки/поля. Кнопка «Опубликовать» неактивна до валидности.

## Flatten-правила (публикация)

Для каждого раунда → для каждой строки → выходная категория:
- `name` = имя банк-категории на момент публикации (snapshot);
- `questions` — по возрастанию `column.value`; каждый: контент резолвится из банк-вопроса
  (`type`/`prompt`/`answer`/`media`), `value` = цена столбца, `special` = тег ячейки;
- media-пути переписываются `bank/media/<file>` → `media/<file>` и файлы **копируются**
  (не move) в `{mediaDir}/{packId}/media/...`.

Результат = существующий формат пака → проходит существующую `gameJsonSchema`. **Движок не
меняется.**

## Экспорт / импорт шаблона (2b-ext)

Сетка + цены + теги + ссылки на вопросы по UID, **без медиа** (оно в банке). Импорт
резолвит ссылки против текущего банка; отсутствующие UID → предупреждение, ячейка
помечается невалидной (требует пере-привязки). Это отдельная ветка переноса, не связанная
с экспортом банка (2a) и не связанная с публикацией.

## UI Конструктора (хэндофф §5.7, эхо `Matrix.svelte`)

- **Экран «Список игр»** (core): создать пустую / из шаблона 5×5 / открыть / удалить /
  (2b-ext) импорт шаблона.
- **Экран игры**: редактируемое **название** (инлайн-инпут) + индикатор автосейва
  («Сохранение… → Сохранено ✓») + вкладки раундов («+ Раунд»). По умолчанию шаблон 5×5,
  авто-цены `100..500 × номер раунда`.
- **Сетка раунда**: заголовки-столбцы с редактируемой ценой (золотые числовые инпуты);
  кнопки **+ / −** столбец; строки-категории, снизу **+ Строка**, у строки — удаление.
  Сетка остаётся прямоугольной при любых +/−.
- **Левая ячейка строки** — зона приёма категории: пусто → пунктир «Перетащите
  категорию»; привязано → имя категории + удалить.
- **Сайдбар «База · источник»**: дерево банк-категорий; категория — draggable-карточка
  (грип-иконка), под ней её вопросы (draggable-чипы с иконкой типа). Перетаскивание
  категории на строку → привязка (`categoryId`); перетаскивание вопроса в ячейку —
  **только в строку своей категории** (в свою → зелёная рамка-подсветка; в чужую →
  красная рамка с иконкой запрета, бросок отклоняется).
- **Ячейка**: превью назначенного вопроса + **тег мини-игры по клику-циклу**
  `нет → Кот → Аукцион → нет` (взаимоисключающе; `special` none/cat/auction); крестик
  очищает ячейку.
- **Лайв-валидация**: подсветка пустых/невалидных ячеек/строк + плашка «**Что мешает
  опубликовать**: строк без категории N · пустых ячеек M»; когда всё заполнено — зелёное
  «**Всё заполнено — можно публиковать**».
- **Панель действий** (шапка): «Скачать шаблон» (2b-ext), «Сыграть тестовую»,
  «Опубликовать».
- **Модалка публикации** (`Modal.svelte`): «Новый пак» / «Перезаписать текущую» / «Отмена».

## «Сыграть тестовую»

publish (`mode:"new"`) → берём `packId` из ответа publish → `POST /api/games`
(`{ packId, title, teamCount }`; `teamCount` — дефолт, т.к. это meta/max — команды ведущий
заводит уже в игре) → открыть `/host?game=<id>` и `/board?game=<id>`.

## Корнер-кейсы целостности (упрощены мягкой инвалидацией)

- **Удаление вопроса/категории из банка**: ничего в банке не блокируем (cascade 2a). В
  играх ячейки/строки с висящим UID становятся невалидными при загрузке шаблона
  (лайв-валидация), публикация блокируется до пере-привязки.
- **Смена `categoryId` у строки игры**: все ячейки строки **очищаются** (с подтверждением
  через `Modal`).
- **Перенос вопроса между категориями банка**: запрещён инвариантом 2a (только
  delete+создать заново).

## Тестирование

Сервер (основной фокус, новые файлы в `src/http/` / `src/packs/`):

- Шаблоны: CRUD; preflight (счёт живых игр по packId из `events`); публикация
  `new`/`overwrite`; strict-валидация нового слоя; flatten корректно мапит цену столбца и
  тег ячейки, порядок вопросов, snapshot имени категории, копирование медиа; export/import
  шаблона (резолв ссылок, предупреждение об отсутствующих UID) — 2b-ext.

Клиент: 1–2 smoke-теста на критичные утилиты — draft-state grid (add/dup/reorder сохраняет
стабильные id), ограничение DnD «вопрос только в свою строку», валидатор игры → strict.

Визуальная проверка через Playwright (по проектному правилу): скриншоты Конструктора и
функц. drag-тесты на живом контейнере.
