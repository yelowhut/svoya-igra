# «Своя игра» — движок (Sub-project 1) — Design Spec

**Дата:** 2026-06-22
**Статус:** утверждён к написанию плана
**Область:** игровой движок (host + players + board), потребляющий ZIP-пак заданного формата.
**Вне области (отдельный спек позже):** визуальный редактор паков.

---

## 1. Цель и контекст

Веб-приложение для игры в «Свою игру» дома с друзьями. Ведущий сидит за
ноутбуком/планшетом, игроки — с мобильных телефонов, общий экран-табло выводится
на ТВ/проектор. Запуск в **локальной сети** (один WiFi/роутер), упаковано в Docker.

Главные нефункциональные требования: быстро, удобно, красиво, легко, и —
критично — **честный антиснайп-замер «кто нажал первым»** и **надёжное
сохранение состояния** с возможностью продолжить после любого сбоя.

### Решённые вопросы (зафиксированы при брейншторме)

- Среда: локальная сеть, один хост. Docker обязателен.
- Сервер физически крутится **на ноуте ведущего** → сценарий восстановления
  после смерти ноута описан в инструкции (поднять контейнер на другой машине
  с тем же volume).
- Стек: Node 20 + TypeScript (сервер и фронт).
- Архитектура: модульный монолит в одном контейнере (не сетевые микросервисы).
- Антиснайп: F1-таймер + штраф за фальстарт = эскалирующий случайный блок кнопки.
- Неверный ответ: −стоимость, ход к следующей команде в зафиксированной очереди.
- Аукцион и «Кот в мешке» — по канону «Своей игры».
- Три роли/экрана: ведущий, игрок, отдельное табло (ответ на табло не выводится).
- Финал MVP: итоговое табло + фанфары/шейм (каноничный «Финал» заложен в формат
  пака на будущее, в MVP не реализуется).
- Право выбора: в начале раунда — команда с минимальным счётом; при общем
  провале вопроса ход остаётся у выбиравшей команды.
- Визуальный стиль: тёмный TV-show неон, крупная типографика, анимации, звук.

## 2. Архитектура

**Один Docker-контейнер**: Node 20 + TypeScript. Встроенный **SQLite**
(`better-sqlite3` — синхронный, durable, один файл) на Docker volume.
Брокеры/Redis не используются: на масштабе нескольких команд внутрипроцессный
pub/sub быстрее и проще. «Микросервисность» трактуется как чистые внутренние
модульные границы, без сетевых хопов на горячем пути таймера.

### Внутренние модули (каждый тестируется изолированно)

| Модуль | Ответственность | Зависимости |
|---|---|---|
| `engine` | Игровой автомат, правила, подсчёт очков. Чистая логика, без I/O. | — |
| `buzzer` | Ранжирование нажатий по реакции, фальстарты, блокировки. Чистая логика. | — |
| `packs` | Импорт/валидация ZIP-пака (zod), распаковка медиа. | — |
| `persistence` | Event log + снэпшоты в SQLite, восстановление состояния. | SQLite |
| `realtime` | WebSocket-шлюз (Socket.IO), комнаты, слой сессии, реконнект. | engine, persistence |
| `http` | Раздача статики, загрузка пака, отдача медиа. | packs |

**Граница:** `engine` и `buzzer` — чистые функции/редьюсеры над состоянием,
не знают про сеть и БД. Это делает их полностью покрываемыми юнит-тестами и
переносимыми. `realtime`/`persistence` — адаптеры вокруг них.

### Фронтенд

Один Vite-проект на **Svelte** (крошечный бандл — под цель «лёгкий»). Три точки
входа с lazy-загрузкой:

- `/host` — ведущий
- `/play` — игрок (минимальный бандл на телефон)
- `/board` — табло (ТВ/проектор)

> Отвергнутые альтернативы: React (тяжелее бандл), сетевые микросервисы (вредят
> точности таймера и усложняют ops), Redis (избыточен на этом масштабе).
> Если предпочтителен React — заменяемо без изменения серверной части.

## 3. Роли и экраны

**Ведущий (`/host`):** матрица категорий/стоимостей, клик по вопросу, текст
вопроса крупно, **ответ мелким шрифтом внизу**, кнопки ВЕРНО/НЕВЕРНО, управление
фазами и таймером, очередь buzzer, ручная правка счёта, **undo последнего
вердикта**, индикатор connected/disconnected по каждому игроку.

**Игрок (`/play`, телефон):** ввод Фамилии и Имени (**оба обязательны**),
создание/выбор команды, большая кнопка ОТВЕТ, своя позиция в очереди,
уведомления (+очки / «ВЫБИРАЙТЕ ВОПРОС» / «ВЫ ОТВЕЧАЕТЕ»).

**Табло (`/board`, ТВ/проектор):** матрица, текущий вопрос, очередь команд,
табло очков, анимации фаз и звук. **Правильный ответ на табло не выводится
никогда.**

До создания игры ведущим `/play` и `/board` показывают плейсхолдер
«Игра ещё не началась».

## 4. Buzzer — антиснайп (ядро системы)

Модель F1-таймера, нечувствительная к сетевому джиттеру:

1. Ведущий читает вопрос, жмёт «Зарядить». Идёт обратный отсчёт со **случайной
   задержкой** до сигнала GO (предугадать момент нельзя).
2. Сервер шлёт `GO` с серверным временем. Каждый клиент **локально** фиксирует
   `T_получения` (момент приёма GO) и `T_нажатия`, и отправляет на сервер
   **только число `reaction = T_нажатия − T_получения`** (миллисекунды), а не
   таймстемп. Это убирает влияние задержек/джиттера сети: «секундомер» у каждого
   клиента запускается локально.
3. Сервер ранжирует команды по возрастанию `reaction`.
4. **Анти-чит:** `reaction < ~100 мс` физиологически нереально для человека →
   трактуется как фальстарт. Порог вынести в конфиг.
5. **Фальстарт** (нажатие до GO или подозрительно быстрая реакция): игроку
   случайный блок кнопки **0.5–0.7 с**; при рецидиве длительность ×2, ×4, …
   Во время блока легко пропустить GO — спамить кнопку невыгодно и рискованно.
6. **Очередь даётся КОМАНДЕ:** засчитывается только **первое валидное нажатие
   любого участника** команды; последующие нажатия той же команды игнорируются.
   Итог — упорядоченная очередь команд: команда₁ → команда₂ → …

Очередь видна у ведущего, у игроков и на табло.

> Замечание: ранжирование по `reaction`, а **не** по времени прихода сообщения
> на сервер. Замер по приходу был бы несправедлив — выиграла бы команда ближе к
> точке доступа (WiFi-джиттер телефонов 5–50 мс перекрывает реальную разницу
> реакций).

## 5. Игровой цикл и правила

### Автомат фаз

```
LOBBY → ROUND_INTRO → PICKING → QUESTION → BUZZER_ARMED → BUZZER_OPEN
      → ANSWERING → JUDGED → (PICKING | ROUND_END) → … → GAME_END
```

### Обычный вопрос

- **Верно** → команде **+стоимость**; эта команда выбирает следующий вопрос.
- **Неверно** → команде **−стоимость**; ход к следующей команде в зафиксированной
  очереди buzzer.
- **Очередь исчерпана / таймаут / никто не нажал** → ведущий открывает ответ,
  очков никому, **право выбора остаётся у выбиравшей команды**.
- **Начало раунда** → выбирает **команда с минимальным счётом** (при равенстве —
  ведущий назначает).

### Аукцион («Вопрос со ставкой»)

- Торг по очереди команд; ставка **≥ номинала** ячейки и **≤ текущих очков**
  команды.
- Победитель торга отвечает **один**, без buzzer.
- Верно → **+ставка**; неверно → **−ставка**.

### Кот в мешке

- Выбравшая команда **обязана передать** вопрос другой команде (не себе).
- Назначенная команда отвечает **одна**, без buzzer.
- Стоимость берётся из пака и **скрыта от получателя** до вердикта.

### Конец раунда / игры

- Конец раунда → экран позиций команд.
- Конец игры → итоговое табло, **фанфары победителям, шейм проигравшим**.

## 6. Состояние, персистентность, реконнект (критично)

### Три разных артефакта (не смешивать)

- **Pack** — содержимое игры: вопросы, ответы, медиа, тип ячейки
  (обычная/аукцион/кот). Импортируется из ZIP, **иммутабелен** после импорта.
- **GameConfig** — конфигурация партии: ссылка на Pack + название игры + число
  команд + порядок раундов. Это и есть **«Saved»-шаблон** для переигровки.
- **GameSnapshot** — живое состояние партии: счёт команд, использованные клетки,
  текущая фаза, текущий вопрос, очередь, лог событий.

### Event sourcing

Каждое значимое действие → **append-only событие** в SQLite. **Снэпшот** полного
состояния раз в N событий. Восстановление = загрузить последний снэпшот + реплей
хвоста событий. Переживает рестарт сервера, потерю питания, краш процесса.
Бонус — чистый аудит-лог партии.

События **идемпотентны**: повторная доставка одного и того же действия (напр.
после реконнекта) не двоит счёт и не ломает состояние (по уникальному id события).

### Слой сессии и реконнект

Надёжность обеспечивает **отдельный слой сессии**, живущий независимо от сокета.
Серверная сессия игрока хранится в стейте игры и привязана к `clientToken`
(UUID), **а не к `socketId`**. Сокет — лишь текущая «труба», может пересоздаваться
произвольно.

Сценарии:

1. **Блокировка экрана / сворачивание вкладки.** Сокет рвётся (детект по
   heartbeat), игрок помечается `disconnected`, **место в команде, имя и позиция
   сохраняются**. При возврате на событиях `visibilitychange`/`pageshow` клиент
   инициирует реконнект, шлёт `clientToken`, сервер привязывает новый сокет к
   существующей сессии. Действий от игрока не требуется.
2. **Закрытие браузера/страницы.** JS-контекст уничтожен, транспортный
   реконнект бессилен — но `clientToken` лежит в **localStorage** и переживает
   закрытие. Игрок снова открывает URL комнаты (на табло и у ведущего — **QR-код**),
   на загрузке выполняется автоматический `rejoin` по токену; **имя вводить не
   надо**.
3. **Краш/разрядка/новый телефон/очистка браузера.** localStorage потерян. Игрок
   открывает URL, заново вводит имя и **выбирает свою команду**. Потеря личной
   идентичности почти ничего не стоит: **очки начисляются КОМАНДЕ, а не игроку**,
   счёт команды цел. Опционально: при совпадении имени в команде предложить
   «это снова ты?» и переиспользовать слот.

Дополнительно:
- **Screen Wake Lock API** — пока у команды ход или buzzer заряжен, удерживаем
  экран активным (не панацея — ОС может проигнорировать, поэтому слой сессии
  обязателен в любом случае).
- **Индикатор у ведущего** — видно `connected/disconnected` по каждому игроку.
- **Дисконнект во время buzzer** не критичен: за команду жмёт любой участник.
  **Дисконнект во время ответа** не критичен: ответ устный в комнате, ведущий
  судит как обычно.

### Смерть ноута ведущего (сервер живёт там же)

В инструкции: поднять контейнер на другой машине в сети, примонтировав volume с
файлом SQLite. Игра продолжится с последнего сохранённого события. Клиенты
переподключаются на новый адрес хоста.

## 7. Формат пака (ZIP) и доставка контента

`pack.zip`:
- `game.json` — структура: `rounds[] → categories[] → questions[]`. У вопроса:
  ```
  {
    type: "text" | "image" | "audio",
    prompt: string,        // текст вопроса
    media?: string,        // путь внутри media/ (для image/audio)
    answer: string,        // правильный ответ (виден только ведущему)
    value: number,         // стоимость (тир)
    special: "none" | "auction" | "cat"
  }
  ```
- `media/` — папка с картинками и аудио.

Импорт валидируется через **zod** с человекочитаемыми ошибками. В состав первого
спека входят:
- **детальная дока по ручной сборке пака**;
- **готовый пример-пак** для немедленного локального теста.

Каноничный «Финал» в формате зарезервирован (поле/тип на будущее), в MVP не
реализуется.

---

## 8. Транспорт и реалтайм

**Socket.IO** — комнаты (per-game, per-role) и быстрый автореконнект транзиентных
обрывов из коробки. Чуть тяжелее голого `ws`, но надёжность важнее на телефонах с
прыгающим WiFi.

> Явно: устойчивость к **закрытию страницы/блокировке** обеспечивает **слой
> сессии** (раздел 6), а не транспорт. Socket.IO закрывает только транзиентные
> обрывы при живом JS-контексте.

---

## 9. Обработка ошибок

- Валидация пака при импорте с понятными сообщениями (что именно не так и где).
- Идемпотентные события: повторная доставка не двоит счёт/состояние.
- **Host-undo** последнего вердикта (откат +/− стоимости и фазы).
- Защита от двойного засчитывания нажатия команды.
- Сетевые обрывы обрабатываются слоем сессии (раздел 6).

## 10. Тестирование

TDD. Покрытие:

- **`engine`** (юнит): переходы автомата, подсчёт очков (верно/неверно), правила
  аукциона, правила «кота в мешке», право выбора (мин. счёт / провал вопроса).
- **`buzzer`** (юнит): ранжирование по реакции, отсечение фальстартов по порогу,
  эскалация блока, **дедуп по команде** (первое валидное нажатие).
- **`packs`** (юнит): валидация валидного и битого `game.json`.
- **Интеграция:** реконнект (token → rejoin, сохранение места), восстановление
  состояния из event log (снэпшот + реплей).

## 11. Визуальный стиль

Тёмный TV-show неон: глубокий тёмный фон, яркие неоновые акценты, крупная
типографика, анимации фаз и звуковое сопровождение (особенно на табло).
Детальная проработка — на этапе реализации фронтенда.
