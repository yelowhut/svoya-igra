# SP3 «Лендинг + запуск игры» — дизайн

> Под-проект 3 платформы «Своя игра». Цель: публичный лендинг для игроков на корне
> порта и перенос всего опыта ведущего (запуск + ведение игры) в admin-оболочку
> «Студия» за паролем. `/host` ретайрится. Движок (reducer/типы/event-store) **не
> трогаем** — это явная граница скоупа; механика «Таймер ответа» вынесена в отдельный
> engine-спек, в SP3 под неё только зарезервировано поле в UI/данных.

Дата: 2026-06-23. Стек и текущее состояние — см. memory `svoya-igra-status` и
`docs/ui-design-brief.md`. Дизайн-эталон — `docs/design_handoff_svoya_igra/`
(прототип `Своя игра - Студия.dc.html`, токены README §8, скриншоты §5.1/5.5/5.6).

---

## 1. Скоуп

**Входит:**
1. **Публичный лендинг** на `/` — показывает единственную активную игру (или приглашение).
2. **Перенос ведущего в admin-shell** — «Лобби и команды» и «Пульт · игра» становятся
   реальными секциями рейла «Студия»; `/host` ретайрится.
3. **Запуск из опубликованных паков** — выбор готовой игры из списка `packs` вместо
   загрузки `.zip`.
4. **Модель активации** — явная «Активировать»; ровно одна активная игра; только она
   видна на лендинге.

**Не входит (отдельные спеки):**
- Движковая механика «Таймер ответа» (TIMEOUT в reducer, серверно-авторитетный отсчёт,
  отображение отсчёта). В SP3 — только зарезервированное поле и визуальный контрол без
  живой логики.
- Экспорт/импорт шаблона игры (2b-ext), «Финальный раунд» (2c).
- Удаление игр (только «Завершить игру»).

## 2. Архитектура: роутинг и web-входы

| URL | Было | Стало |
|---|---|---|
| `/` | fallback → `index.html` (host) | **Лендинг** (публичный) |
| `/play`, `/play?game=…` | экран игрока | без изменений |
| `/board?game=…` | табло | без изменений |
| `/admin`, `/admin/*` | admin-shell (Конструктор/База) | + секции «Лобби», «Пульт» |
| `/host` | host-экран | **удаляется** (серверный роут `app.get('/host', …)` снят) |

- `web/index.html` репурпозится: указывает на новый `web/src/landing/main.ts`. Fastify
  static уже отдаёт `index.html` на корне — **новый серверный роут для `/` не нужен**.
- `web/vite.config.ts`: входы `host/play/board/admin` → `landing/play/board/admin`
  (вход `host` уходит, появляется `landing`).
- Каталог `web/src/host/` после декомпозиции (см. §6) удаляется.
- Удаление роута `/host` — сигнал, что он действительно ушёл. Ссылки игрока/табло не
  меняются. `docs/run.md` обновляется (адреса, флоу запуска).
- **`/play` без `?game=`** (старый публичный список комнат через `GET /api/games`) —
  редиректит на `/` (лендинг). Это снимает зависимость экрана игрока от `/api/games` и
  позволяет перевести `GET /api/games` за `requireAdmin` (см. §4) без поломки игрока.
  Вход игрока остаётся `/play?game=<id>` (бриф §3B).

## 3. Модель данных

**Активная игра — синглтон-указатель (подход A, движок не трогаем).**

Новая таблица:
```sql
CREATE TABLE IF NOT EXISTS active_game (
  id           INTEGER PRIMARY KEY CHECK (id = 1),
  game_id      TEXT,            -- NULL = нет активной игры
  activated_at INTEGER
);
```
Репозиторий `src/persistence/activeGameRepo.ts`:
- `setActiveGame(gameId)` — upsert строки id=1, перезаписывает `game_id` (гарантия «одна
  активная»: новая активация снимает предыдущую).
- `clearActiveGame()` — `game_id = NULL`.
- `clearActiveGameIfMatches(gameId)` — очистить, только если активна именно эта игра.
- `getActiveGameId(): string | null`.

Обоснование выбора синглтона вместо события `GAME_ACTIVATED`: «видимость на лендинге» —
мета-флаг UI, а не доменное событие истории партии. Событие задело бы `reducer.ts` /
`types.ts` / event-store — те же файлы, что позже заденет таймер-спек, и нарушило бы
границу «движок не трогаем». Синглтон не засоряет реплей и тривиально сбрасывается.

**Зарезервированное поле таймера.** В настройках запуска UI хранит выбранное «Время на
ответ» (сек). В SP3 значение принимается и сохраняется (например, в `meta` создаваемой
игры через существующий путь `GAME_CREATED`-payload, **без** изменения reducer-логики —
поле просто проносится), но нигде не применяется. Если пронос без правки reducer
невозможен без касания типов — храним значение клиентски/в `active_game`-смежной мете и
помечаем TODO для engine-спека. Решение по месту хранения фиксируется на этапе плана; в
любом случае **reducer не меняется**.

## 4. API-контракт

Все админ-эндпоинты — за `requireAdmin` (кука `svoya_admin`). Socket.io host-actions
остаются **открытыми** (LAN-доверие) — правка socket-хендшейка вне скоупа.

| Метод | Путь | Гард | Назначение |
|---|---|---|---|
| `GET` | `/api/active-game` | публичный | Лендинг: активная игра или `null` |
| `GET` | `/api/packs` | requireAdmin | Список опубликованных паков для запуска |
| `POST` | `/api/games` | requireAdmin | Создать игру (packId + teamCount + reserved timer) |
| `POST` | `/api/games/:id/activate` | requireAdmin | Сделать игру активной (снять предыдущую) |
| `POST` | `/api/games/:id/deactivate` | requireAdmin | Снять активность (если активна эта) |
| `GET` | `/api/games` | requireAdmin* | Admin-список игр (существует; добавить гард) |
| `GET` | `/api/games/:id/teams` | существует | Лобби игрока (без изменений) |

`*` — `GET /api/games` сейчас открыт; переводится за `requireAdmin` (admin-only список).
Публичному лендингу он не нужен — хватает `/api/active-game`; экран игрока тоже больше не
зависит от него (`/play` без `?game=` редиректит на `/`, см. §2).

**`GET /api/active-game`** — полный набор из брифа §3A:
```jsonc
// нет активной игры:
null
// есть:
{
  "gameId": "…",
  "title": "Пятничный квиз",
  "phase": "PICKING",          // фаза из GameState
  "teamCount": 6,              // макс. команд (meta/max)
  "playerCount": 4,            // активных (connected) игроков
  "totalRounds": 3,            // pack.rounds.length
  "currentRound": 1            // roundIndex + 1
}
```
Источники: `teamCount`, `roundIndex`, `players` — из `loadState(gameId)`; `totalRounds` —
из пака игры. Если активная игра в `GAME_END` — указатель уже очищен (см. ниже), поэтому
эндпоинт вернёт `null` (или приглушённый архив — решается на этапе UI, по умолчанию `null`).

**`GET /api/packs`** → `[{ id, title, rounds }]` из таблицы `packs`.

**Авто-деактивация при `GAME_END`.** В точке, где сервер эмитит/обрабатывает переход в
`GAME_END` (force-завершение при перезаписи пака уже существует в `templates.ts`; плюс
завершение по ходу игры), вызывается `clearActiveGameIfMatches(gameId)`. Лендинг никогда
не показывает завершённую игру как доступную для входа.

**`.zip`-upload** убирается из UI запуска. Эндпоинт `POST /api/packs` (upload) либо
остаётся для совместимости за `requireAdmin`, либо удаляется — решается на этапе плана;
импорт банка в редакторе (2a) не затрагивается.

## 5. Экраны

### 5.1. Лендинг — `/` (игроки, телефон/ноутбук)

Новый Vite-вход `web/src/landing/` (App.svelte + main.ts) на токенах «Студия»
(`theme.css`), мобайл-фёрст, **без admin-рейла** (рейл в прототипе — навигатор демо).

- **Данные:** `GET /api/active-game` на `onMount` + **поллинг** ~3–5 с (без отдельного
  публичного socket-канала) — оживляет `playerCount` / `phase` / `currentRound`.
- **Точное следование §5.1 прототипа:**
  - Карточка-герой с radial-glow в шапке; заголовок дисплейной гарнитурой (Oswald).
  - **Есть игра:** статус «Идёт сейчас · раунд N из M» (зелёная точка-glow), название,
    метрики (команд из макс. / игроков / раундов), кнопки **«Войти в игру»** (primary
    `#7c5cff`) → `/play?game=<id>` и **«Открыть табло»** (outline) → `/board?game=<id>`.
  - **Пусто:** пунктирная карточка-приглашение «Пока нет активной игры — ведущий вот-вот
    её запустит» (тон приглашения, не ошибка).
  - **Завершена/архив:** при необходимости приглушённая карточка (opacity .62); по
    умолчанию завершённая игра уже снята с активной → состояние «Пусто».
  - **Загрузка:** скелет/«Загружаем…».

### 5.2. Лобби и команды — `/admin/lobby` (ноутбук ведущего)

Декомпозиция setup- и LOBBY-фазы `host/App.svelte`. Точное следование §5.5 прототипа.

- **Создание игры:** выбор пака из `GET /api/packs` (вместо `.zip`), число команд,
  **«Время на ответ»** — степпер ±5 и пресеты **30 / 45 / 60** (визуально как в
  прототипе; значение — в зарезервированное поле, **логика не подключена**). → `POST
  /api/games`.
- **Список игр** (`GET /api/games`) — продолжить/выбрать рабочую игру (резюм ведущего
  через localStorage сохраняется).
- **Шапка:** «название · N команд · K игроков · M раундов» + **«Начать раунд 1»**.
- **Команды:** список (цветная точка, переименовать/удалить пустую, «+ Команда»).
- **Ростер:** игроки по командам, онлайн/офлайн, зона переноса (drag&drop между
  командами) — перенос существующей механики.
- **«Активировать»** (`POST …/activate`) — игра появляется на лендинге; игроки заходят
  уже в лобби-фазе (бриф §3A). Индикатор «Активна — видна игрокам». «Начать игру» →
  `hostAction('startRound', { roundIndex: 0 })`.

### 5.3. Пульт · игра — `/admin/pult` (ноутбук ведущего)

Декомпозиция фаз PICKING…JUDGED / ROUND_END / GAME_END. Точное следование §5.6 прототипа.

- Читает рабочую/активную игру из admin-store (`/admin/pult` **без** URL-параметра —
  «одна активная» это гарантирует).
- **Левая колонка:** сетка выбора вопроса (кликабельна в фазе выбора; отыгранные гаснут;
  выбранная — фиолетовая обводка; иконки мини-игр) + карточка вопроса (категория+цена,
  текст, **ответ только ведущему** в золотой пунктирной плашке; контролы Кот/Аукцион).
- **Правая колонка (рейл «Ход»):** по фазе — «Открыть buzzer» / баннер «Отвечает
  <Команда>» + большие **✓ Верно** / **✕ Неверно** / «Следующий вопрос»; очередь buzzer
  (номер/команда/дельта); правка счёта ±100 (отвечающая подсвечена).
- **Чип «Ответ 45 с»** и кружок отсчёта — визуально как в прототипе, **статично** (живой
  отсчёт — engine-спек).
- **Быстрая деактивация** + **«Завершить игру»** (эмитит `GAME_END` → авто-очистка
  указателя). Удаления игр в MVP нет.

### Общий слой
```
web/src/admin/gameSocket.ts   (НОВЫЙ) — socket-подписка на gameState + hostAction(),
                                переиспользуется Лобби и Пультом
web/src/admin/store.ts        — + workingGameId / activeGameId
web/src/admin/Rail.svelte     — включить пункты «Лобби и команды» / «Пульт · игра»
                                (снять disabled, добавить aria-current — backlog 2-shell)
```

## 7. Состояния и тон (бриф §4, прототип §6.8)

- Лендинг: есть игра / пусто (приглашение) / завершена (архив) / загрузка.
- Лобби/Пульт: резюм прерванной игры; офлайн-игроки (тускло, тег «офлайн»); ошибки
  запуска (пак не найден) — в голосе интерфейса, что не так и что делать.
- Кнопки активным глаголом, имя действия сохраняется через флоу.

## 8. Тестирование и верификация

**Серверные (vitest, корень):**
- `activeGameRepo`: set/clear/get, перезапись (одна активная), `clearIfMatches`,
  авто-очистка при `GAME_END`.
- API: `activate`/`deactivate` (401 без куки), `GET /api/active-game` (null / полный
  объект с метриками), `GET /api/packs` (за гардом), `GET /api/games` (теперь за гардом).
- Регрессия: host-actions через socket не сломаны; force-завершение при перезаписи пака
  чистит указатель.

**Web unit (vitest из `web/`):** маппинг `active-game` → карточка лендинга; `gameSocket`
после выноса (подписка/hostAction).

**Playwright E2E:** для быстрых dev-прогонов допускается изолированный :3100, но
**обязательный финальный гейт — на пересобранном и задеплоенном полноценном приложении**:
- `docker compose up -d --build`, рантайм на **:3000**.
- Полный флоу: admin-логин → Лобби (выбор пака из списка, создать игру, команды) →
  «Активировать» → лендинг `/` показывает игру с метриками → Пульт (выбрать вопрос,
  открыть buzzer, судейство) → «Завершить игру» → лендинг очистился.
- Лендинг: нет активной → приглашение; активна → карточка + «Войти в игру».
- `/host` ретайрнут (404/редирект); `/play`, `/board` целы.
- **Попиксельная сверка** скриншотов Лендинга/Лобби/Пульта против эталонов дизайна
  (`docs/design_handoff_svoya_igra/screenshots/01-landing.png`, `05-lobby.png`,
  `06-host-answering.png`): токены, раскладка, сигнатурные элементы. **Расхождения
  фиксируются в отчёте** (что отличается, насколько, план устранения).

## 9. Открытые решения для этапа плана

- Точное место хранения зарезервированного значения таймера (proned через
  `GAME_CREATED`-payload без правки reducer / отдельная мета) — **reducer не трогаем**.
- Судьба `POST /api/packs` (upload): оставить за гардом для совместимости или удалить.
- Поведение лендинга для только что завершённой игры (по умолчанию `null` → «Пусто»;
  опционально короткий «архив»).
